# Plugin & Workflow System for TaskDock

**Date:** 2026-02-05
**Status:** Draft

## Overview

Add a plugin and workflow system to TaskDock that enables users to extend the application with custom tabs, hook into built-in tabs, and automate multi-step async operations. Plugins are authored as simple file-based packages that can be generated by LLMs from natural language instructions.

### Design Goals

- **LLM-generatable** - Plugin format is simple and declarative enough that an AI tool (Claude Code, Copilot CLI, etc.) can reliably generate correct plugins from natural language descriptions
- **Extensible UI** - Plugins can add new tabs and inject buttons/panels into built-in tabs (PR Review, PR Home, Work Items, Terminals)
- **Async workflow chaining** - Workflows are TypeScript scripts that chain long-running operations (wait for build, call AI, act on response)
- **No sandboxing** - Trusted developer desktop tool; no restrictions on shell or HTTP access
- **Native look and feel** - Plugin UIs use a declarative component catalog that inherits TaskDock's design system

### Motivating Examples

1. **Incident management plugin** - Adds a new Incidents tab that polls for ICM incidents, displays them in a master-detail view, and auto-runs AI analysis on new incidents
2. **PR build-and-review workflow** - A button in the PR tab that waits for a build to complete, sends results to Claude/Copilot, and uses the AI response to take next actions

## Plugin Structure

A plugin is a folder in `~/.taskdock/plugins/<plugin-name>/` containing:

```
my-incident-plugin/
├── manifest.json        # Metadata, config schema, triggers, hooks
├── ui.json              # Component catalog UI definition
├── workflows/           # TypeScript workflow scripts
│   ├── run-analysis.ts
│   ├── poll-incidents.ts
│   └── daily-summary.ts
└── README.md            # For humans and LLMs
```

### manifest.json

Ties together metadata, configuration, triggers, permissions, and hooks:

```jsonc
{
  "id": "incident-manager",
  "name": "Incident Manager",
  "version": "1.0.0",
  "description": "Monitor and analyze ICM incidents with AI",

  "config": {
    "icmEndpoint": { "type": "string", "label": "ICM API Endpoint", "required": true },
    "teamId": { "type": "number", "label": "Team ID" },
    "pollInterval": { "type": "number", "label": "Poll interval (seconds)", "default": 30 },
    "apiToken": { "type": "string", "label": "API Token", "secret": true }
  },

  "triggers": [
    { "type": "manual", "id": "runAnalysis", "workflow": "workflows/run-analysis.ts", "label": "Run Analysis" },
    { "type": "polling", "id": "pollIncidents", "workflow": "workflows/poll-incidents.ts", "interval": "{{config.pollInterval}}s" },
    { "type": "scheduled", "id": "dailySummary", "workflow": "workflows/daily-summary.ts", "cron": "0 9 * * 1-5" }
  ],

  "hooks": {
    "pr-review": {
      "toolbar": [
        { "label": "Check Build & Review", "icon": "play", "trigger": "buildAndReview", "position": "right" }
      ]
    },
    "pr-home": {
      "row-actions": [
        { "label": "Cross-ref Incidents", "icon": "alert-triangle", "trigger": "crossRefIncidents" }
      ]
    }
  }
}
```

## Component Catalog

Plugins compose UI from these pre-built components. All components inherit TaskDock's CSS variables for native look and feel across dark/light themes.

| Component | Key Props | Description |
|---|---|---|
| `table` | `columns`, `dataSource`, `onRowClick`, `polling`, `sortable`, `filterable` | Sortable, filterable data grid with row selection |
| `detail-panel` | `sections`, `dataSource` | Shows detail for selected item |
| `card` | `label`, `content`, `renderAs` (text/markdown/code) | Content block with label |
| `split-panel` | `sizes`, `direction` (horizontal/vertical), `children` | Resizable side-by-side layout |
| `button-group` | `buttons` (label, icon, action, variant) | Action buttons wired to triggers |
| `status-badge` | `value`, `colorMap` | Colored status indicator |
| `key-value` | `dataSource`, `fields` | Label-value pair list |
| `timeline` | `dataSource`, `fields` (time, title, description) | Chronological event list |
| `tabs` | `items` (label, content) | Sub-tab navigation |
| `form` | `fields`, `onSubmit` | Input form for configuration or actions |
| `markdown` | `content` | Rendered markdown block |
| `empty-state` | `icon`, `title`, `description`, `action` | Shown when no data |
| `header` | `title`, `subtitle` | Section header |

### Data Binding

Components reference data through `dataSource` strings that map to workflow outputs. When a workflow calls `ctx.ui.update('incident-list', data)`, the table with `id: "incident-list"` re-renders with that data.

### Actions

Buttons reference trigger IDs from the manifest. When clicked, TaskDock invokes the corresponding workflow with the current selection as `ctx.input`.

## Workflow Scripts & the ctx SDK

Each workflow is a TypeScript file exporting a default async function that receives a `PluginContext` object.

### Example: Incident Analysis

```ts
// workflows/run-analysis.ts
import { PluginContext } from '@taskdock/plugin-sdk';

export default async function(ctx: PluginContext) {
  const { incidentId } = ctx.input;

  // Fetch incident details
  const incident = await ctx.http.get(`${ctx.config.icmEndpoint}/incidents/${incidentId}`);

  // Get impacted services
  const impact = await ctx.http.get(`${ctx.config.icmEndpoint}/incidents/${incidentId}/impact`);

  // Ask AI to analyze
  const analysis = await ctx.ai.claude(
    `Analyze this incident and suggest mitigation steps:\n` +
    `Title: ${incident.title}\n` +
    `Severity: ${incident.severity}\n` +
    `Impact: ${JSON.stringify(impact)}\n` +
    `Description: ${incident.description}`
  );

  // Store result and update UI
  await ctx.store.set(`analysis:${incidentId}`, analysis);
  await ctx.ui.update('incident-detail', { aiAnalysis: analysis });
  await ctx.ui.toast('Analysis complete', 'success');
}
```

### Example: Polling for New Incidents

```ts
// workflows/poll-incidents.ts
import { PluginContext } from '@taskdock/plugin-sdk';

export default async function(ctx: PluginContext) {
  const incidents = await ctx.http.get(
    `${ctx.config.icmEndpoint}/teams/${ctx.config.teamId}/incidents`
  );

  // Check for new incidents since last poll
  const lastPoll = await ctx.store.get('lastPollTime') || 0;
  const newIncidents = incidents.filter(i => new Date(i.createdDate).getTime() > lastPoll);

  if (newIncidents.length > 0) {
    // Auto-run analysis on each new incident
    for (const incident of newIncidents) {
      await ctx.run('runAnalysis', { incidentId: incident.id });
    }
    await ctx.ui.toast(`${newIncidents.length} new incident(s) analyzed`, 'info');
  }

  // Update the table and save poll time
  await ctx.ui.update('incident-list', incidents);
  await ctx.store.set('lastPollTime', Date.now());
}
```

### SDK Reference: PluginContext

| Namespace | Methods | Description |
|---|---|---|
| `ctx.ado` | `getPR()`, `postComment()`, `getWorkItems()`, `updateStatus()` | Azure DevOps operations using existing ADO API service |
| `ctx.ai` | `claude(prompt)`, `copilot(prompt)` | AI provider calls, returns string response |
| `ctx.http` | `get(url, opts?)`, `post(url, body, opts?)`, `put()`, `delete()` | Generic HTTP calls for external APIs |
| `ctx.shell` | `run(command, opts?)` | Run shell commands/scripts, returns stdout/stderr |
| `ctx.ui` | `update(componentId, data)`, `toast(msg, level)`, `inject(tab, location, component)` | Update plugin UI, show notifications, inject into core tabs |
| `ctx.store` | `get(key)`, `set(key, value)`, `delete(key)` | Persistent key-value storage per plugin |
| `ctx.events` | `on(event, callback)`, `emit(event, data)` | Subscribe to and emit events |
| `ctx.config` | Direct property access | Plugin configuration values set by user in Settings |
| `ctx.log` | `info(msg)`, `warn(msg)`, `error(msg)`, `debug(msg)` | Structured logging, visible in plugin log panel |
| `ctx.run` | `run(triggerId, input)` | Invoke another workflow within the same plugin |
| `ctx.input` | Direct property access | Data passed from the trigger (button click payload, etc.) |

## Hooks: Injecting into Built-in Tabs

Plugins can inject buttons and panels into built-in TaskDock tabs via the `hooks` field in `manifest.json`.

### Available Hook Points

| Tab | Hook Location | Context Passed as `ctx.input` |
|---|---|---|
| `pr-review` | `toolbar`, `file-context-menu`, `comments-toolbar`, `bottom-panel` | Full PR data, selected file, comments |
| `pr-home` | `toolbar`, `row-actions` | PR summary |
| `workitems` | `toolbar`, `row-actions` | Work item data |
| `terminals` | `toolbar` | Active terminal session |

### Hook Declaration

```jsonc
{
  "hooks": {
    "pr-review": {
      "toolbar": [
        { "label": "Check Build & Review", "icon": "play", "trigger": "buildAndReview", "position": "right" }
      ],
      "file-context-menu": [
        { "label": "Security Scan File", "icon": "shield", "trigger": "scanFile" }
      ]
    },
    "pr-home": {
      "row-actions": [
        { "label": "Analyze with ICM", "icon": "alert-triangle", "trigger": "crossRefIncidents" }
      ]
    }
  }
}
```

### Context Passed to Hooked Workflows

```ts
// For a pr-review toolbar hook, ctx.input contains:
{
  source: "pr-review",
  pr: { id, title, repository, sourceBranch, targetBranch, ... },
  files: [{ path, changeType, ... }],
  selectedFile: { path, content, diff }  // if a file is selected
}

// For a pr-home row-action hook:
{
  source: "pr-home",
  pr: { id, title, repository, status, ... }
}
```

### Injecting Results Back into Core Tabs

Workflows can render components directly inside built-in tabs using `ctx.ui.inject`:

```ts
export default async function(ctx: PluginContext) {
  const buildStatus = await ctx.shell.run(
    `az pipelines runs list --pipeline-id ${ctx.input.pr.buildPipelineId} --top 1`
  );

  // Show result inline in the PR tab as a collapsible panel
  await ctx.ui.inject('pr-review', 'bottom-panel', {
    type: 'card',
    label: 'Build & Review Status',
    content: buildStatus.summary,
    renderAs: 'markdown'
  });
}
```

## Triggers

Three trigger types supported at launch:

### Manual

Fired when a user clicks a button in the plugin tab or a hooked button in a built-in tab.

```jsonc
{ "type": "manual", "id": "runAnalysis", "workflow": "workflows/run-analysis.ts", "label": "Run Analysis" }
```

### Polling

Runs a workflow at a fixed interval. Interval can reference config values.

```jsonc
{ "type": "polling", "id": "pollIncidents", "workflow": "workflows/poll-incidents.ts", "interval": "{{config.pollInterval}}s" }
```

### Scheduled

Runs a workflow on a cron schedule.

```jsonc
{ "type": "scheduled", "id": "dailySummary", "workflow": "workflows/daily-summary.ts", "cron": "0 9 * * 1-5" }
```

## Plugin Engine Architecture

The plugin engine lives in the Node.js backend sidecar, alongside existing services.

```
┌─────────────────────────────────────┐
│          Renderer (Frontend)         │
│                                      │
│  ┌──────────┐  ┌──────────────────┐ │
│  │ Core Tabs │  │ Plugin Tabs      │ │
│  │ (PR, WI)  │  │ (rendered from   │ │
│  │           │  │  ui.json catalog)│ │
│  └──────────┘  └──────────────────┘ │
│          │              │            │
└──────────┼──────────────┼────────────┘
           │   WebSocket RPC (5198)
┌──────────┼──────────────┼────────────┐
│          ▼              ▼            │
│  ┌──────────┐  ┌──────────────────┐ │
│  │ Core      │  │ Plugin Engine    │ │
│  │ Services  │  │                  │ │
│  │           │  │ - Loader         │ │
│  │ (ADO,AI,  │  │ - Scheduler      │ │
│  │  Terminal) │  │ - Script Runner  │ │
│  │           │  │ - ctx Builder    │ │
│  └──────────┘  └──────────────────┘ │
│       Backend Bridge (Node.js)       │
└──────────────────────────────────────┘
```

### Plugin Engine Components

1. **Loader** - Scans `~/.taskdock/plugins/`, validates each `manifest.json` against the plugin schema, registers tabs and triggers on startup. Watches for file changes and hot-reloads plugins.

2. **Scheduler** - Manages polling and cron triggers. Uses `setInterval` for polling, a lightweight cron parser for scheduled triggers. Starts/stops triggers with the plugin lifecycle.

3. **Script Runner** - Executes workflow `.ts` files using `tsx` (already a project dependency). Each execution gets its own `ctx` instance. Runs in a child worker to avoid blocking the main backend event loop.

4. **ctx Builder** - Constructs the `PluginContext` object for each workflow execution, wiring `ctx.ado` to the existing ADO API service, `ctx.ai` to existing AI providers, `ctx.http` to fetch, `ctx.shell` to child_process, etc.

---

## Plugin Lifecycle

| Stage | Action |
|---|---|
| **Install** | Drop plugin folder into `~/.taskdock/plugins/` |
| **Configure** | Plugin appears in Settings with config fields from `manifest.config` |
| **Activate** | Plugin tab appears in sidebar, polling/scheduled triggers start |
| **Disable** | User toggles off in Settings, all triggers stop, tab hidden |
| **Uninstall** | Delete the plugin folder |

Hot-reload: When any file in a plugin folder changes, the plugin is automatically reloaded without restarting TaskDock.

---

## Error Handling

- **Try/catch wrapper** - Every workflow execution is wrapped automatically. Uncaught errors are caught, logged, and shown as a toast with a "View Log" action.
- **Plugin log panel** - Each plugin gets a collapsible log section at the bottom of its tab showing recent executions, errors, and `ctx.log` output.
- **Timeout** - Workflows have a default 60s timeout (configurable per-trigger in manifest via `"timeout": 120`). Prevents runaway scripts.
- **Retry** - Polling/scheduled workflows that fail are retried once after a short delay. If they fail again, the trigger is paused and the user is notified.

## LLM Plugin Generation

For AI tools to generate plugins correctly, TaskDock provides reference materials in `~/.taskdock/plugins/_schema/`:

### Files Provided

| File | Purpose |
|---|---|
| `plugin-schema.json` | JSON Schema for `manifest.json` and `ui.json` validation |
| `plugin-sdk.d.ts` | TypeScript type definitions for `PluginContext` |
| `README.md` | Overview of the plugin format for LLM context |

### Example Plugins

Bundled in `~/.taskdock/plugins/_examples/`:

- `incident-manager/` - ICM monitoring with polling and AI analysis
- `pr-build-watcher/` - PR build wait + AI review chain
- `deployment-tracker/` - Simple table + scheduled daily summary

### Generation Flow

1. User tells their AI tool: *"Create a TaskDock plugin that monitors ICM incidents for team X and runs AI analysis on new ones"*
2. The AI reads the schema, examples, and SDK types from the `_schema/` folder
3. It generates the plugin folder with `manifest.json`, `ui.json`, and workflow scripts
4. TaskDock hot-reloads the plugin, user sees the new tab immediately
5. If something's wrong, user asks the AI to fix it and iterates

No special tooling required. Any AI tool that can read and write files works.

## Implementation Scope

### Phase 1: Core Plugin Engine
- Plugin loader with manifest validation
- Script runner using `tsx`
- `ctx` SDK with all namespaces (`ado`, `ai`, `http`, `shell`, `ui`, `store`, `config`, `log`, `run`)
- Manual triggers (button clicks)
- Plugin settings UI (config fields, enable/disable)

### Phase 2: UI Rendering
- Component catalog renderer (table, card, split-panel, button-group, status-badge, key-value, detail-panel)
- Plugin tab registration in sidebar
- Data binding (`ctx.ui.update` pushes data to components)
- Toast notifications from plugins

### Phase 3: Hooks & Injection
- Hook point registration in built-in tabs (PR Review, PR Home, Work Items, Terminals)
- `ctx.ui.inject` for rendering plugin components inside core tabs
- Context passing from core tabs to hooked workflows

### Phase 4: Scheduling & Polish
- Polling triggers
- Scheduled/cron triggers
- Hot-reload file watcher
- Plugin log panel
- Error retry logic
- Schema, SDK types, and example plugins in `_schema/` and `_examples/`
